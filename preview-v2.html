核心理念：预览区 = 信息仪表盘，不是终端截图

Wave Terminal 的 Block

每次命令的输入+输出封装为独立块，预览显示"命令 + 结果摘要"，而非原始输出流。

参考: wave.dev

GitHub Actions 的 Job 卡片

不显示原始日志，而是显示结构化步骤：步骤名 + 状态图标 + 耗时，点击展开详情。

参考: github.com/actions

Docker Desktop 的容器卡片

显示结构化信息（状态、端口、CPU/内存），而非 docker logs 原始输出。

参考: docker.com/desktop

PM2 的进程列表

进程名 + 状态 + CPU/内存 + 运行时长，一目了然，不需要看原始 stdout。

参考: pm2.keymetrics.io

Grafana 的仪表盘

关键指标 + 趋势图 + 状态灯，把海量数据提炼为几个核心数字。

参考: grafana.com

VS Code 的 Problems 面板

不显示编译器原始输出，而是提取 error/warning 结构化展示：文件 + 行号 + 消息。

参考: code.visualstudio.com
方案 A: 命令 + 结构化摘要
推荐
核心思路：从终端输出中提取"最后执行的命令"和"结果摘要"，用结构化方式展示。
信息来源：利用已有的 statusDetector 规则引擎，提取命令提示符后的文本作为"当前命令"， 提取输出中的关键数据（成功/失败、数字指标、文件列表等）作为摘要。
参考：Wave Terminal 的 Block 概念 + GitHub Actions 的 Job 卡片

方案 B: 活动流 / 时间线
创新
核心思路：把终端活动抽象为"事件时间线"，每个事件是一行结构化信息（时间 + 类型 + 摘要）。
信息来源：从输出流中提取关键事件：命令执行、成功/失败、状态变化、用户交互等。
参考：CI/CD Pipeline 的 Step 列表 + 日志查看器的时间戳导航

方案 C: 指标面板 + 尾部输出
实用
核心思路：上半部分是"指标面板"（运行时长、输出行数、错误数、最后命令），下半部分是精简后的尾部输出。
信息来源：指标从 session 元数据中提取，尾部输出保留 ANSI 颜色但做智能裁剪（去控制序列、去空行）。
参考：Docker Desktop 容器卡片 + PM2 进程列表

方案 D: 智能提取（命令 + 结果 + 关键数据）
创新
核心思路：把预览区分为三个区域：顶部显示"最后命令"，中间显示"结构化结果"（自动从输出中提取关键数据），底部显示"精简尾部"。
信息来源：命令从 prompt 模式匹配中提取；结果通过正则规则引擎提取（构建时间、文件数、错误信息等）；尾部保留 ANSI 颜色。
参考：VS Code Problems 面板 + Wave Terminal Block + GitHub Actions Step

方案对比

维度 方案 A: 命令+摘要 方案 B: 活动流 方案 C: 指标+尾部 方案 D: 智能提取
信息密度 ★★★★★ ★★★ ★★★★ ★★★★★
一眼看懂 ★★★★★ ★★★★ ★★★ ★★★★
实现难度 中等 — 需要命令提取+摘要规则 高 — 需要事件流解析 低 — 指标已有，尾部复用现有逻辑 高 — 需要智能提取引擎
性能影响 低 — 只处理尾部 中 — 需要维护事件列表 低 — 指标从元数据计算 中 — 规则匹配有开销
适用场景 通用，特别适合构建/部署 长时间运行的任务 需要快速扫状态 复杂 CLI 工具输出
参考产品 Wave Terminal, GitHub Actions CI/CD Pipeline, 日志查看器 Docker Desktop, PM2 VS Code Problems, Wave Block
