# 卡片排序逻辑梳理与优化建议

## 一、当前排序逻辑

### 1.1 排序规则概览

排序逻辑位于 `App.tsx` 第 396-420 行，采用多级排序策略：

```
第一优先级：会话状态类型
├─ 有状态（error/needs-confirm/needs-input）→ 排最前
└─ 无状态（running/idle）→ 排在后面

第二优先级：状态严重程度（仅在有状态内部）
├─ error（优先级 0）→ 最前
├─ needs-confirm（优先级 1）→ 中间
└─ needs-input（优先级 2）→ 最后

第三优先级：稳定活动时间（仅在同类型状态内部）
├─ stableActivityAt 倒序（最近活动的在前）
├─ createdAt 正序（创建早的在前）
└─ id 字典序（兜底）
```

### 1.2 核心时间戳字段

#### `lastActivityAt`（最后活动时间）
- **更新时机**：每次有新输出时更新（`flushSession` 函数中）
- **作用**：用于判断会话是否空闲
- **代码位置**：
  ```typescript
  // App.tsx flushSession 函数
  state.updateSession(sessionId, {
    history: newHistory,
    previewText: cleanText,
    status: 'running',
    lastActivityAt: Date.now()  // ← 每次输出都更新
  })
  ```

#### `stableActivityAt`（稳定活动时间）
- **更新时机**：仅在会话从 `running` 转为 `idle` 时更新
- **作用**：用于排序，避免频繁输出导致卡片抖动
- **代码位置**：
  ```typescript
  // App.tsx 空闲检测定时器
  if (s.status === 'running' && now - s.lastActivityAt > IDLE_THRESHOLD_MS) {
    state.updateSession(s.id, {
      status: 'idle',
      stableActivityAt: now  // ← 仅在转为 idle 时更新
    })
  }
  ```

#### `IDLE_THRESHOLD_MS`（空闲阈值）
- **值**：10000 毫秒（10 秒）
- **含义**：超过 10 秒没有新输出，会话从 `running` 转为 `idle`

### 1.3 状态转换流程

```
会话创建
  ↓
status = 'running'
lastActivityAt = 创建时间
stableActivityAt = 创建时间
  ↓
有新输出
  ↓
lastActivityAt = 当前时间
status 保持 'running'（如果之前是 idle，会转回 running）
  ↓
10秒无输出
  ↓
status = 'idle'
stableActivityAt = 当前时间  ← 关键：此时更新稳定时间
  ↓
又有新输出
  ↓
status = 'running'
lastActivityAt = 当前时间
stableActivityAt 保持不变  ← 关键：不会更新
```

### 1.4 排序示例

假设有 5 个会话：

| 会话 | 状态 | stableActivityAt | createdAt | 排序结果 |
|------|------|------------------|-----------|----------|
| A | error | 1000 | 500 | 1（有状态，error 最严重） |
| B | needs-confirm | 1200 | 600 | 2（有状态，needs-confirm 次之） |
| C | needs-input | 1100 | 700 | 3（有状态，needs-input 最后） |
| D | running | 900 | 400 | 4（无状态，stableActivityAt 最大） |
| E | idle | 800 | 300 | 5（无状态，stableActivityAt 最小） |

**排序过程**：
1. 先按状态类型分组：[A, B, C] 和 [D, E]
2. 有状态组内按严重程度：A(error) → B(needs-confirm) → C(needs-input)
3. 无状态组内按 stableActivityAt 倒序：D(900) → E(800)
4. 最终顺序：A → B → C → D → E

## 二、潜在问题分析

### 2.1 问题 1：running 状态的排序不稳定

**现象**：
- 会话 A 和 B 都是 `running` 状态
- A 的 `stableActivityAt` 是 1000，B 是 900
- A 排在 B 前面
- 但是 A 和 B 都在持续输出，`lastActivityAt` 不断更新
- 由于 `stableActivityAt` 不变，排序保持稳定 ✓

**但是**：
- 如果 A 停止输出 10 秒，转为 `idle`
- A 的 `stableActivityAt` 更新为当前时间（比如 2000）
- 此时 A 的 `stableActivityAt` (2000) > B 的 `stableActivityAt` (900)
- A 仍然排在 B 前面 ✓

**结论**：当前逻辑在 `running` 状态下是稳定的，因为 `stableActivityAt` 不会频繁更新。

### 2.2 问题 2：idle 状态的排序可能不符合预期

**现象**：
- 会话 A 在 10:00 转为 `idle`，`stableActivityAt` = 1000
- 会话 B 在 10:05 转为 `idle`，`stableActivityAt` = 1300
- B 排在 A 前面（因为 1300 > 1000）

**问题**：
- B 比 A 更晚空闲，但排在 A 前面
- 用户可能期望"最近活跃的排前面"，但 B 已经空闲了
- 如果 B 之前一直在输出，A 已经空闲很久，B 应该排在 A 前面 ✓

**结论**：当前逻辑符合"最近活跃的排前面"的预期。

### 2.3 问题 3：新创建的会话排序位置

**现象**：
- 会话 A 在 10:00 创建，`stableActivityAt` = 1000
- 会话 B 在 10:05 创建，`stableActivityAt` = 1300
- B 排在 A 前面（因为 1300 > 1000）

**问题**：
- 新创建的会话总是排在前面（因为 `stableActivityAt` 更大）
- 如果用户有很多会话，新创建的会话会不断"跳"到前面
- 这可能导致视觉上的不稳定

**但是**：
- 新创建的会话通常有输出，处于 `running` 状态
- `running` 状态的会话排在 `idle` 前面
- 所以新创建的会话排在前面是合理的 ✓

**结论**：当前逻辑在新建会话场景下是合理的。

### 2.4 问题 4：有状态会话的时间排序

**现象**：
- 会话 A 在 10:00 触发 `error`，`stableActivityAt` = 1000
- 会话 B 在 10:05 触发 `error`，`stableActivityAt` = 1300
- A 和 B 都是 `error` 状态
- 按严重程度排序，A 和 B 优先级相同
- 按 `stableActivityAt` 排序，B 排在 A 前面（1300 > 1000）

**问题**：
- 有状态会话的排序也依赖 `stableActivityAt`
- 但 `stableActivityAt` 只在转为 `idle` 时更新
- 如果 A 和 B 一直是 `error` 状态，`stableActivityAt` 不会更新
- 排序会稳定在初始值

**潜在问题**：
- 如果 A 在 10:00 触发 `error`，`stableActivityAt` = 1000
- A 在 10:02 转为 `idle`，`stableActivityAt` = 1120
- A 在 10:03 又触发 `error`，`stableActivityAt` 保持 1120
- 此时 A 的 `stableActivityAt` 是 1120，而不是 1000
- 这可能导致排序不符合预期

**结论**：有状态会话的排序可能因为 `stableActivityAt` 的历史更新而变得复杂。

### 2.5 问题 5：缺少用户自定义排序

**现象**：
- 当前排序逻辑是固定的，用户无法自定义
- 用户可能希望按名称、创建时间、分组等排序
- 当前只能通过分组和状态筛选来间接控制排序

**结论**：缺少灵活的排序选项。

## 三、优化建议

### 3.1 建议 1：优化有状态会话的排序

**问题**：有状态会话的 `stableActivityAt` 可能在状态转换过程中被更新，导致排序不稳定。

**方案**：
- 有状态会话按 `lastActivityAt` 排序（最近触发状态的排前面）
- 无状态会话按 `stableActivityAt` 排序（最近空闲的排前面）

**实现**：
```typescript
return filtered.sort((a, b) => {
  const aHas = hasStatus(a.status)
  const bHas = hasStatus(b.status)
  
  if (aHas && bHas) {
    // 两组都有状态：先按严重程度
    const diff = statusPriority(a.status) - statusPriority(b.status)
    if (diff !== 0) return diff
    // 严重程度相同：按 lastActivityAt 倒序（最近触发状态的排前面）
    if (a.lastActivityAt !== b.lastActivityAt) {
      return b.lastActivityAt - a.lastActivityAt
    }
  }
  
  if (aHas !== bHas) {
    // 有状态的排前面
    return aHas ? -1 : 1
  }
  
  // 同为无状态：按 stableActivityAt 倒序
  if (a.stableActivityAt !== b.stableActivityAt) {
    return b.stableActivityAt - a.stableActivityAt
  }
  
  // 兜底排序
  if (a.createdAt !== b.createdAt) return a.createdAt - b.createdAt
  return a.id.localeCompare(b.id)
})
```

**优点**：
- 有状态会话按最近触发时间排序，更符合"紧急处理"的预期
- 无状态会话按稳定时间排序，避免抖动

**缺点**：
- `lastActivityAt` 频繁更新可能导致有状态会话排序不稳定
- 但可以通过防抖或节流来缓解

### 3.2 建议 2：添加用户自定义排序选项

**方案**：
- 在工具栏添加排序选项下拉框
- 支持按以下字段排序：
  - 状态优先级（默认）
  - 创建时间
  - 名称
  - 最后活动时间

**实现**：
```typescript
// 在 store 中添加排序字段
interface AppState {
  sortBy: 'status' | 'createdAt' | 'name' | 'lastActivityAt'
  setSortBy: (sortBy: 'status' | 'createdAt' | 'name' | 'lastActivityAt') => void
}

// 在 App.tsx 中根据 sortBy 选择排序逻辑
const sortSessions = (sessions: Session[]) => {
  switch (sortBy) {
    case 'status':
      // 当前排序逻辑
      return sessions.sort(/* ... */)
    case 'createdAt':
      return sessions.sort((a, b) => b.createdAt - a.createdAt)
    case 'name':
      return sessions.sort((a, b) => a.name.localeCompare(b.name))
    case 'lastActivityAt':
      return sessions.sort((a, b) => b.lastActivityAt - a.lastActivityAt)
    default:
      return sessions
  }
}
```

**优点**：
- 用户可以根据需求选择排序方式
- 提高灵活性

**缺点**：
- 增加 UI 复杂度
- 需要持久化用户选择

### 3.3 建议 3：优化空闲检测逻辑

**问题**：
- 当前空闲检测每秒执行一次，遍历所有会话
- 如果会话数量很多（100+），可能造成性能问题

**方案**：
- 使用事件驱动而非轮询
- 每次更新 `lastActivityAt` 时，设置一个 10 秒的定时器
- 定时器触发时，检查是否真的空闲，然后更新状态

**实现**：
```typescript
// 在 flushSession 中
const updateLastActivity = (sessionId: string) => {
  const state = useAppStore.getState()
  const session = state.sessions.find(s => s.id === sessionId)
  if (!session) return
  
  // 更新 lastActivityAt
  state.updateSession(sessionId, {
    lastActivityAt: Date.now(),
    status: 'running'
  })
  
  // 清除之前的定时器
  if (idleTimers[sessionId]) {
    clearTimeout(idleTimers[sessionId])
  }
  
  // 设置新的定时器
  idleTimers[sessionId] = setTimeout(() => {
    const currentState = useAppStore.getState()
    const currentSession = currentState.sessions.find(s => s.id === sessionId)
    if (currentSession && currentSession.status === 'running') {
      currentState.updateSession(sessionId, {
        status: 'idle',
        stableActivityAt: Date.now()
      })
    }
  }, IDLE_THRESHOLD_MS)
}
```

**优点**：
- 减少不必要的遍历
- 更精确的空闲检测

**缺点**：
- 需要管理定时器生命周期
- 增加代码复杂度

### 3.4 建议 4：添加排序动画

**问题**：
- 排序变化时，卡片位置突然改变，视觉上不流畅

**方案**：
- 使用 CSS 过渡动画
- 卡片位置变化时平滑移动

**实现**：
```css
.session-card {
  transition: transform 0.3s ease;
}
```

**优点**：
- 提升用户体验
- 减少视觉突兀感

**缺点**：
- 可能影响性能（大量卡片时）
- 需要调整动画参数

### 3.5 建议 5：支持拖拽排序

**方案**：
- 允许用户拖拽卡片自定义顺序
- 保存用户自定义顺序到 store
- 提供"恢复默认排序"选项

**实现**：
```typescript
// 在 Session 类型中添加 order 字段
interface Session {
  // ... 其他字段
  customOrder?: number  // 用户自定义排序
}

// 在 store 中添加排序方法
const reorderSessions = (fromIndex: number, toIndex: number) => {
  const state = useAppStore.getState()
  const sessions = [...state.sessions]
  const [removed] = sessions.splice(fromIndex, 1)
  sessions.splice(toIndex, 0, removed)
  
  // 更新 customOrder
  sessions.forEach((s, i) => {
    state.updateSession(s.id, { customOrder: i })
  })
}
```

**优点**：
- 最灵活的排序方式
- 用户完全控制

**缺点**：
- 实现复杂（需要拖拽库）
- 需要处理与自动排序的冲突

## 四、推荐方案

### 短期优化（1-2 天）

1. **优化有状态会话排序**（建议 1）
   - 有状态会话按 `lastActivityAt` 排序
   - 无状态会话保持 `stableActivityAt` 排序
   - 工作量：0.5 天

2. **添加排序动画**（建议 4）
   - 使用 CSS 过渡动画
   - 工作量：0.5 天

### 中期优化（1 周）

3. **添加用户自定义排序选项**（建议 2）
   - 在工具栏添加排序下拉框
   - 支持 4 种排序方式
   - 工作量：2 天

4. **优化空闲检测逻辑**（建议 3）
   - 使用事件驱动替代轮询
   - 工作量：2 天

### 长期优化（2-3 周）

5. **支持拖拽排序**（建议 5）
   - 集成拖拽库（如 react-dnd）
   - 处理与自动排序的冲突
   - 工作量：5 天

## 五、总结

当前排序逻辑整体是合理的，核心设计思路清晰：
- 有状态会话优先，按严重程度排序
- 无状态会话按稳定活动时间排序
- 使用 `stableActivityAt` 避免抖动

主要问题：
1. 有状态会话的排序可能因为 `stableActivityAt` 的历史更新而变得复杂
2. 缺少用户自定义排序选项
3. 空闲检测使用轮询，可能有性能问题

推荐优先实施：
1. 优化有状态会话排序（按 `lastActivityAt`）
2. 添加排序动画
3. 添加用户自定义排序选项

这些优化可以显著提升用户体验，同时保持代码的简洁性。
